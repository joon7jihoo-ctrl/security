01 — Traffic Flow

네트워크 트래픽 흐름도

인터넷에서 내부 서버까지 패킷이 통과하는 각 장비의 계층과 역할을 인라인(Inline) 배치 기준으로 설명합니다. 가시화 장비와 SIEM은 아웃오브패스(Out-of-Path) 방식으로 트래픽을 미러링하여 수집합니다.

💡
인라인 vs 아웃오브패스 DDoS·방화벽·IPS·WAF는 인라인(Inline)으로 트래픽 경로에 직렬 배치되어 실시간 차단이 가능합니다. 가시화 장비와 SIEM은 아웃오브패스(Out-of-Path)로 트래픽을 미러링 수신하여 서비스 영향 없이 분석합니다.

장비별 계층 및 처리 방식 요약

02 — DDoS Defense

DDoS 방어 시스템

대용량 트래픽(Volumetric) 공격과 자원 고갈(Resource Exhaustion) 공격을 ISP 업스트림 단계에서 차단합니다.

🌊
DDoS Protection
공격 유형별 탐지 및 완화 메커니즘
L3 / L4 / L7 복합 계층 방어
Volumetric Protocol Attack Application Layer BGP Blackhole Scrubbing Center

DDoS 방어 시스템은 인터넷 회선 입구에서 가장 먼저 트래픽을 처리합니다. ISP와 BGP(Border Gateway Protocol) 피어링을 통해 공격 트래픽을 스크러빙 센터(Scrubbing Center)로 우회(Divert)시키거나, 인라인 장비에서 직접 완화합니다.

03 — Firewall

방화벽 (Firewall)

L3/L4 기반 상태 기반(Stateful) 패킷 검사와 접근 제어 정책으로 허가된 세션만 내부 구간으로 전달합니다.

🔥
Next-Generation Firewall
Stateful Inspection & Zone-Based Policy
L3 (IP) / L4 (TCP/UDP) 계층

Stateful Inspection은 단순히 패킷 헤더만 보는 것이 아니라 세션 상태 테이블(State Table)을 유지합니다. 최초 SYN → SYN-ACK → ACK 핸드셰이크를 추적하여, 기존 세션에 속하지 않는 비정상 패킷(예: 갑작스러운 ACK)을 차단합니다.

04 — Intrusion Prevention System

IPS (침입방지시스템)

Deep Packet Inspection(DPI)으로 패킷 페이로드를 시그니처와 비교하여 CVE 기반 취약점 공격을 인라인 차단합니다.

🔬
Intrusion Prevention
DPI 기반 시그니처 매칭 & 이상 탐지
L4 (TCP/UDP) ~ L7 (Application Payload) 계층

IPS는 방화벽이 허용한 세션 안의 페이로드(Payload)까지 검사합니다. 방화벽이 "포트 443이 열렸는지"를 본다면, IPS는 "그 443 포트로 넘어오는 데이터 내용이 Exploit 패턴인지"를 검사합니다.

Deep Packet Inspection 시그니처 DB 프로토콜 이상 탐지 Rate-Based 탐지 False Positive 튜닝

탐지 방식 비교

탐지 방식원리장점단점대표 예시
시그니처 기반알려진 공격 패턴(Regex/Hex) 매칭정확도 높음, 오탐 낮음Zero-Day 탐지 불가CVE-2021-44228 (Log4Shell)
프로토콜 이상RFC 표준 위반 패킷 탐지신규 공격 일부 탐지정상 트래픽 오탐 가능비정상 HTTP 메서드
Rate-Based단위 시간 내 임계치 초과 탐지스캔/브루트포스 탐지정상 대량 트래픽 오탐포트 스캔, 로그인 브루트포스
행위 기반정상 베이스라인 대비 편차 탐지Zero-Day 일부 탐지베이스라인 구축 필요비정상 데이터 전송량

IPS 탐지 이벤트 처리 절차

Step 01 — Detect
이벤트 수신 및 심각도 분류
IPS 콘솔에서 이벤트 수신. Critical / High / Medium / Low / Info 5단계로 심각도 구분. Critical·High는 즉각 분석 대상.
Step 02 — Analyze
시그니처 ID 및 출발지 분석
탐지된 시그니처 ID로 CVE 번호 확인. 출발지 IP의 GeoIP, 평판(Reputation) 조회. 패킷 캡처(PCAP) 확보 후 페이로드 직접 분석.
Step 03 — Validate
False Positive 여부 판단
동일 시그니처가 정상 업무 트래픽에도 매칭되는지 확인. 오탐이면 예외(Exclusion) 처리하고 시그니처 튜닝. 실제 공격이면 차단 적용.
Step 04 — Respond
인라인 차단 또는 SOAR 연계
인라인 차단(Drop/RST) 적용 또는 eyecloudxoar에 이벤트 전달하여 Playbook 자동 실행. IP 차단, 담당자 알림, 티켓 생성.
🚨
False Positive 관리 IPS 시그니처는 오탐 시 정상 업무 트래픽을 차단할 수 있습니다. 신규 시그니처 적용 시 첫 48시간은 Alert 모드로 운용하고, 오탐 여부 확인 후 Block 모드로 전환하는 것이 권장됩니다.
05 — Web Application Firewall

WAF (웹 방화벽)

HTTP/HTTPS 프로토콜을 완전 파싱하여 OWASP Top 10을 포함한 웹 애플리케이션 공격을 L7 계층에서 방어합니다.

🕸️
Web Application Firewall
HTTP 전체 파싱 & OWASP Top 10 방어
L7 (HTTP / HTTPS / WebSocket) 계층 — Reverse Proxy 배치

WAF는 HTTP 요청의 URL, 쿼리 파라미터, 헤더, 쿠키, 요청 바디 전체를 파싱합니다. HTTPS 트래픽은 WAF에서 복호화(SSL Offloading) 후 검사하고 재암호화하여 서버로 전달합니다. 일반적으로 리버스 프록시(Reverse Proxy) 형태로 배치됩니다.

WAF 동작 모드

모드 01
Passive (감지만)
트래픽을 차단하지 않고 탐지 이벤트만 기록. 신규 배포 시 오탐 분석용.
모드 02
Alert Mode
탐지 이벤트를 SIEM으로 전송하지만 차단하지 않음. 정책 튜닝 기간.
모드 03
Block Mode
탐지된 요청 즉시 차단 + 403 응답 반환. 운영 환경 표준 모드.
모드 04
Challenge Mode
CAPTCHA / JS Challenge로 사람 여부 검증. 봇 트래픽 대응.

07 — SIEM / SOAR

eyecloudxoar (SIEM/SOAR)

이기종 보안 장비의 로그를 수집·정규화하고, 상관관계 분석(Correlation)으로 위협을 식별한 뒤 Playbook 기반 자동 대응까지 단일 콘솔에서 수행합니다.

⚡
eyecloudxoar Platform
통합 보안관제 플랫폼 — SIEM + SOAR
전 계층 로그 집계 — 상관관계 분석 — 오케스트레이션

eyecloudxoar는 SIEM(보안 정보 이벤트 관리)과 SOAR(보안 오케스트레이션·자동화·대응) 기능을 통합한 플랫폼입니다. 장비별로 포맷이 다른 로그를 CEF/LEEF/JSON으로 정규화하고, 상관관계 규칙으로 단일 이벤트로는 보이지 않는 복합 공격 패턴을 식별합니다.

로그 정규화 (Normalization) 상관관계 분석 (Correlation) 위협 인텔리전스 (TI) Playbook 자동화 티켓팅 (Ticketing) 대시보드

08 — SOC Pipeline

보안관제 파이프라인 프로세스

탐지 이벤트가 eyecloudxoar에 집계되어 티켓팅 및 Playbook 자동 대응으로 이어지는 전체 워크플로우입니다.

Playbook 예시

🔴 PB-001: 외부 IP 브루트포스 대응
Critical · 자동
1
트리거: IPS에서 동일 src_ip 로그인 실패 ≥ 15회 / 30초 탐지 AUTO
2
위협 인텔리전스 조회: src_ip를 TI 피드(VirusTotal, AbuseIPDB) 자동 조회 → 악성 IP 여부 확인 AUTO
3
자동 차단: 방화벽 API 호출 → src_ip 차단 규칙 즉시 추가 (24시간 유효) AUTO
4
티켓 생성: eyecloudxoar에 인시던트 티켓 자동 생성, 담당자 SMS 발송 AUTO
5
관제원 검토: 담당자가 오탐 여부 확인 후 차단 유지 또는 해제 결정 (수동)
6
사후 처리: 티켓 종결, 처리 내역 기록, 주간 보고서 반영
🟠 PB-002: 웹 공격 (SQLi / XSS) 대응
High · 반자동
1
트리거: WAF에서 동일 src_ip의 SQLi/XSS 탐지 ≥ 5회 / 10분 AUTO
2
패킷 캡처: 해당 세션 PCAP 자동 수집 및 eyecloudxoar 티켓에 첨부 AUTO
3
IP 임시 차단: WAF에서 해당 IP 30분 차단 적용 AUTO
4
페이로드 분석: 관제원이 PCAP 검토 → 실제 공격 여부, 취약점 노출 여부 확인 (수동)
5
취약점 대응: 실제 취약점 존재 시 개발팀에 패치 요청 티켓 연동 (수동)
